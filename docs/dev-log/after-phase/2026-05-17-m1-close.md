### Substantive M1 code changes (the two R/ touches in M1)

1. **M1.6 — `R/extract-repeatability.R`** (PR [#154](https://github.com/itchyshin/gllvmTMB/pull/154)).
   One-line `vW` correction adding `link_residual_per_trait(fit)`
   to the within-tier variance:

   ```r
   sigma2_d <- unname(link_residual_per_trait(fit))
   ...
   vW <- diag(Lambda_W %*% t(Lambda_W)) + sd_W^2 + sigma2_d
   ```

   Headline audit-finding from M1.1: the pre-fix formula was
   `vW <- diag(Lambda_W %*% t(Lambda_W)) + sd_W^2`, which
   silently dropped the per-family link residual in non-Gaussian
   repeatability denominators. Backward compat: Gaussian rows
   add a zero (no behaviour change). Walks
   **MIX-06: partial / silent-bug → covered.**

2. **M1.8 — `R/methods-gllvmTMB.R` + `R/fit-multi.R` + `R/bootstrap-sigma.R`**
   (PR [#157](https://github.com/itchyshin/gllvmTMB/pull/157)).
   The substantive M1 fix:

   - `.draw_y_per_family()` (~100 lines): per-row dispatch by
     `family_id_vec` (0 = gaussian, 1 = binomial, 2 = poisson,
     3 = lognormal, 4 = Gamma, 5 = nbinom2; other families warn
     once and fall back to Gaussian-on-link-scale).
   - `fit$family_input`: preserves the original family list
     (with `family_var` attribute) before the line 174
     `family <- family[[1]]` truncation; `fit$family` continues
     to be the first family for downstream `linkinv` use.
   - `bootstrap-sigma.R`: `refit_one` reads `fit$family_input`
     when available, falling back to `fit$family` for pre-M1.8
     fits.

   Walks **MIX-08: partial → covered** AND **MIS-05: partial →
   covered**. The M1.4 follow-up symptom (degenerate ±1
   bootstrap CIs on rank-1 latent mixed-family fits) is also
   resolved as a consequence.

## 8. What Did Not Go Smoothly

Three points captured during M1 worth carrying forward:

- **Cascading bug discovery in M1.4 → M1.8** (Curie + Fisher).
  The original M1.4 implementation surfaced degenerate ±1
  bootstrap CIs on rank-1 latent mixed-family fits. First
  hypothesis (link_residual propagation gap) was wrong; root
  cause was the *cascading* pair of bugs documented in
  M1.8's after-task §4: (a) `simulate.gllvmTMB_multi` applied
  Gaussian noise to ALL rows regardless of per-row family, and
  (b) `fit$family <- family[[1]]` at `R/fit-multi.R:174`
  truncated mixed-family lists to one family. Both surfaced
  only when the bootstrap resample fed back through `simulate`
  with the truncated family. **Lesson**: when a single
  symptom has multiple plausible root-cause hypotheses, file
  the audit doc *before* committing to a fix — pulling MIS-05
  into M1.8 was the right call only because the audit walked
  both candidates before the code went in.

- **M1.2 fixture identifiability dance** (Curie + Boole). The
  M1.2 fixture went through three revisions mid-PR before
  ratifying the three-tier design: small (T=3, d=1) for test
  speed, medium (T=8, d=2) for demonstration density, future
  (T≥20, d≥3) for the post-M1 showcase. Each revision
  surfaced a different identifiability concern (parameter-
  counting boundary, group-wise mean-centred casts that
  collapsed Gamma/nbinom2 traits to constants, C-locale factor-
  sort divergence under withr::local). **Lesson**: fixture
  design is hard; multiple identifiability concerns can stack;
  pre-M1.3 audit work paid for the M1.3 → M1.8 test stability.

- **Internal helper exposure deferred to follow-up**
  (Pat + Rose). The M1.9 article uses
  `gllvmTMB:::load_mixed_family_fixture()` — an `@keywords
  internal` + `@noRd` helper. User-facing articles using `:::`
  is a documented stop-gap; a small post-M1 follow-up PR
  exposes the fixture loader as `@export` (or as a `data()`-
  style accessor). Logged as a future PR; not blocking M1.10.
