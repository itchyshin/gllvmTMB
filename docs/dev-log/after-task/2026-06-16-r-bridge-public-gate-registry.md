# After Task: Public Julia Bridge Gate Registry

**Branch**: `codex/r-bridge-grouped-dispersion`
**Date**: `2026-06-16`
**Roles (engaged)**: Ada, Hopper, Emmy, Rose, Shannon, Grace

## 1. Goal

Make `GJL-GATE-*` errors inspectable from R. Users and maintainers should be
able to see a gate id, status, source, reason, issue, representative test, and
validation row without reading internal source code or loading Julia.

## 2. Implemented

- Exported `gllvm_julia_gate_registry()` as a read-only wrapper around the
  internal gate registry.
- Added roxygen documentation and the generated
  `man/gllvm_julia_gate_registry.Rd` topic.
- Added the new topic to the pkgdown Julia bridge reference section.
- Added the pre-existing generated `gllvmTMB_julia-methods` topic to
  `_pkgdown.yml` after `pkgdown::check_pkgdown()` caught the missing reference
  row.
- Added NEWS wording that frames the registry as an audit table, not a new
  modelling capability.
- Tightened tests so the public wrapper equals the internal registry and so
  `GJL-GATE-CORRELATION-INTERVALS` maps to `JUL-01A` while
  `GJL-GATE-MASK-X-CI` maps to `JUL-01`.

## 5. Tests of the Tests

The registry test now fails if:

- the public wrapper diverges from the internal registry;
- `GJL-GATE-CORRELATION-INTERVALS` loses its `JUL-01A` extractor-boundary
  mapping;
- `GJL-GATE-MASK-X-CI` is incorrectly assigned to `JUL-01A`.

## 6. Consistency Audit

- `JUL-01` and `JUL-01A` remain `partial`.
- The public registry is read-only and does not admit any new bridge row.
- The NEWS wording says users can distinguish bridge gates from engine limits;
  it does not claim full parity, CRAN readiness, or new interval support.
- `_pkgdown.yml` now lists all generated Julia bridge reference topics.

## 7. Roadmap Tick

This closes another part of the `gllvmTMB#488` gate transparency work: gate ids
are no longer only internal test fixtures. It does not close #488 because the
broader R/Julia landing and full gate-to-matrix audit remain draft.

## 8. What Did Not Go Smoothly

`pkgdown::check_pkgdown()` exposed that `gllvmTMB_julia-methods` had been
generated earlier in the bridge stack but was not in `_pkgdown.yml`. The fix
landed in the same slice because this was the first local pkgdown reference
check after adding the public gate registry topic.

## 9. Team Learning

- Hopper: every gate id should have a user-visible way to inspect why it fired.
- Emmy: S3 method topics generated by roxygen need pkgdown placement just like
  function topics.
- Rose: a read-only registry is claim-safe only if it says "gated" and points
  to validation rows rather than implying support.
- Grace: `pkgdown::check_pkgdown()` was the right local gate for this export.

## 10. Known Limitations And Next Actions

- The registry is flat. It does not yet group gates by family/design row or
  produce a matrix-style maintainer dashboard.
- The public registry does not replace the issue-ledger comments needed before
  closing any bridge governance issue.
